Architecture

System Overview

AML Shield is structured as a pipeline of four cooperating layers: ingestion, screening, risk scoring, and case management. Each layer is independently operable, allowing financial institutions to integrate only the components relevant to their existing compliance stack.

Core Logic

How It Works

1. Ingestion & Normalisation

Transactions and entity records are ingested from upstream systems via batch file or streaming feed. Each record is normalised to a canonical schema — standardising currencies, date formats, name representations, and account identifiers — before being passed downstream. Enrichment at this stage appends jurisdiction, account age, and historical volume statistics.

2. Sanctions Screening

All parties to a transaction are screened against consolidated watchlists (OFAC SDN, UN Consolidated, EU Financial Sanctions). Matching uses a combination of exact lookup and fuzzy string matching to surface near-matches caused by transliteration variations or deliberate obfuscation.

3. Entity Resolution

A recurring challenge in AML is that the same real-world entity appears under multiple names, account numbers, or spellings across systems. AML Shield's entity resolution layer clusters account records using probabilistic matching across name, address, date-of-birth, and transaction-behaviour features — consolidating fragmented profiles before risk scoring.

4. Network Graph Construction

Resolved entities and their transactions are modelled as a directed graph, where nodes are accounts and edges are fund flows. Graph-level features — degree centrality, clustering coefficient, hub-and-spoke patterns, rapid fan-out — are extracted as inputs to the ML model and as standalone typology signals.

5. Risk Scoring Pipeline

Each transaction alert is scored on a 0–100 risk scale by combining two signals:

A
Rule Engine

A set of configurable typology rules encode known money-laundering patterns — structuring (transactions just below reporting thresholds), high-velocity cash cycling, transactions to high-risk jurisdictions. Rules fire as binary flags, each contributing a weighted additive score.

B
ML Anomaly Detector

A supervised model trained on labelled SAR data combines transaction-level features (amount, time-of-day, counterparty risk, currency) with graph-level features extracted from the network layer. SHAP values are computed for every prediction to produce a human-readable explanation of the score.

The two signals are blended via a configurable weighted average, allowing compliance teams to tune the relative contribution of rule-based and statistical detection depending on their regulatory environment and false-positive tolerance.

6. Case Management & SAR Workflow

Alerts above a configurable threshold enter an investigator queue ranked by risk score. The investigator dashboard surfaces the full evidence pack: transaction history, entity profile, network graph visualisation, rule flags, and SHAP explanation. Investigators can escalate to a Suspicious Activity Report, with the platform auto-populating the SAR narrative template from the alert data. Every action is written to an immutable audit trail for regulatory review.
